# DD Monitor v3.51

距离原项目停更已经过去将近一年。v2.26 是当时的紧急修复版本——主播列表刷不出来、部分直播间加载失败、依赖版本不兼容，这些问题让工具几乎处于不可用状态。那个版本的目标很单纯：让它重新跑起来。

v3.51 则完全不同。在确认基本功能恢复之后，我把整个项目的骨架拆开看了一遍，结论是：播放器和弹幕这两块如果不做外科手术，后面只会越修越烂。所以就有了这次改动——底层从 VLC 换到 MPV，弹幕从 ASS 字幕方案换到 OpenGL 叠加渲染，配置管理从一个散落在主窗口各处的"保存按钮"变成一个真正的模块。下面按模块梳理。

---

## 播放器：从 VLC 到 MPV

VLC 的方案需要对每个直播间启动独立的 VLC 实例，再通过窗口捕获嵌入 Qt。这个架构在 Windows 上问题不少：VLC 的 DLL 插件目录臃肿（`plugins/` 目录下有 200 多个 DLL），不同版本的兼容性难以保证，而且 CPU 占用居高不下——16 个窗口同时运行的时候，仅仅是 VLC 实例的管理开销就不小。

v3.51 将整个播放链迁移到了 MPV，通过 `libmpv` 的 render API 直接在 `QOpenGLWidget` 上渲染视频帧。几个关键收益：

- 不再需要安装 VLC，也不需要携带任何 VLC 插件。Windows 用户只需要一个 `libmpv-2.dll` 文件（约 112MB），项目目录瘦身了将近 80MB。
- OpenGL 渲染路径意味着弹幕可以直接用 QPainter 画在视频帧上，不需要依赖 MPV 的字幕系统去做 `sub-reload`——那个方案每次刷新都会导致画面短暂黑屏，多窗口时尤其明显。
- 硬件解码方面，Windows 上由于 libmpv 与 ANGLE 的兼容问题，OpenGL 模式下主动禁用了硬解以规避花屏。如果将来 libmpv 修复了这个问题，可以重新打开。RTX 2060 的软解性能在 16 窗口下完全够用。

画质切换（五档）、音量增强（0.5x - 4.0x）、全局播放控制这些交互逻辑保持不变，底层实现全部重写。

---

## 弹幕系统：三件套重构

弹幕是这次改动最大的部分，也是决定整个项目体验的核心。旧方案是 `MpvDanmakuRenderer`——生成 ASS 字幕文件、通过 `sub-add` 加载为字幕轨道、每次新弹幕到达时 `sub-reload` 刷新。这个方案有两个致命问题：一是频繁的文件写入和字幕重载造成明显的闪烁；二是 ASS 的 `\move()` 标签依赖 libass 的插值，在不同平台上表现不一致。

新的弹幕系统拆成了三个独立模块，各司其职：

### 渲染器 (`danmaku_renderer.py`)

直接在 OpenGL 帧上用 QPainter 绘制弹幕文字。每条弹幕先渲染成 `QImage` 精灵——颜色、描边（宽度可调）、阴影（可开关，强度可调）一次性到位——然后根据布局引擎分配的轨道位置，在 `paintGL()` 里画出来。相比 ASS 方案，这个做法完全消除了文件 I/O 和字幕刷新的开销。

精灵有 LRU 缓存（最多 512 条），相同文字和样式的弹幕只渲染一次。这个设计在处理"888888"这类高频重复弹幕时效果显著——同样内容的弹幕全部命中缓存，几乎零渲染成本。

### 布局引擎 (`danmaku_layout.py`)

滚动弹幕需要分配轨道。每条弹幕从屏幕右侧进入，以固定速度向左移动，经过整个屏幕后消失。如果两条弹幕在同一轨道上时间重叠，就会叠在一起，完全无法阅读。

`RollLayout` 维护一个轨道池，每条轨道记录当前占用者的过期时间。新弹幕到达时优先分配到空闲轨道；如果所有轨道都在使用中，就丢弃这条弹幕（B站自己也是这个策略，业内标准做法）。顶部固定弹幕和底部固定弹幕由 `TopLayout` 和 `BottomLayout` 分别管理，逻辑类似。

轨道的数量由字体大小和显示区域比例自动计算。显示区域比例可以在全局弹幕设置中调节（0.1 - 1.0，即屏幕高度的 10% 到 100%）。

### 设置面板 (`danmu.py`)

新增了 `GlobalDanmuOption`——一个合并了弹幕窗设置和滚动弹幕设置的综合面板。弹幕窗部分控制悬浮弹幕机的透明度、字体大小、横向纵向占比、同传过滤关键词、礼物和进入信息的显示策略。滚动弹幕部分控制字体、速度（50% - 200%）、描边粗细（0 - 6px）、阴影效果、顶部和底部弹幕的开关。

`RollingOptionWidget` 是滚动弹幕的独立设置面板，所有控件都做了范围约束（如描边粗细 0-60，经换算后为 0-6px），避免用户误操作产生极端效果。

每个窗口也保留了独立的弹幕机，可以通过控制栏的「弹」按钮在三种模式之间切换：全开（弹幕机 + 滚动）、仅弹幕机、全关。

## 配置管理

旧版本的配置是典型的"生长型代码"——每次加新功能就多几行 `json.dump`。格式迁移靠临时补丁，保存时机没有控制，用户连续拖几下音量条就能触发十几次磁盘写入。更糟的是，备份逻辑其实是同时写入四份完全相同的文件——如果某次保存写入了坏数据，四份备份一起坏。备份机制形同虚设。

`ConfigManager` 把这些全部收敛到了一个类里：

- 加载时自动做格式迁移，兼容从 v1.x 到 v3.x 的所有配置格式。包括 `roomid` 从 list 到 dict 的迁移、弹幕配置从 bool 到 list 的升级、滚动弹幕默认值的填充、sessionData 的 URL 解码。
- 保存有 500ms 的去抖动（QTimer single-shot），用户连续调整参数时不会触发多次磁盘写入。
- 备份是真正的轮转：写入新配置前，先把当前 `config.json` 挪到 `备份1`，`备份1` 挪到 `备份2`，`备份2` 挪到 `备份3`。这样无论发生什么，至少有一份上一轮的完整配置可以回滚。备份覆盖依赖 `os.rename`（原子操作），不依赖文件拷贝。
- 修复了一个存在已久的 bug：房间列表（`roomid`）在关闭程序后丢失。问题出在 `LiverPanel` 修改了内存中的房间列表但从未回写到 `config` 字典，导致 `configManager.save()` 保存的永远是启动时读到的旧值。修复方案是通过 `dumpConfig` 信号在每次房间变更时将 `roomIDDict` 同步回 `config['roomid']`。

## 登录

废弃了 `webBrowser.py`（基于 Qt WebEngine 的方案，需要加载完整 Chromium 内核，启动慢且内存占用大，甚至不在 `requirements.txt` 里），改用 `login.py` 的 `QRLoginWidget`。原理：调用 B站 API 生成登录二维码，`qrcode` 库渲染为 QPixmap 显示在界面上，后台线程轮询扫码状态。

Session 信息持久化在 `config.json` 中。启动时如果检测到有效的 SESSDATA，自动调 B站 API 验证登录状态。Token 过期前通过 `CredentialRefreshWorker` 自动刷新。登录成功后自动拉取关注列表填充卡片面板。

## 稳定性

这次花了很多精力在"能跑但不对"的问题上：

### 线程安全

`RecordThread`（录制线程）的 `recordToken` 和 `downloadToken` 在跨线程访问时没有任何保护。主线程可以随时把 `recordToken` 设为 `False` 来停止录制，而录制线程在自己的 `run()` 循环里读同一个值。在高频录制场景下，这属于未定义行为。加了 `threading.Lock`，所有跨线程访问都走 `with self._lock`。同时提供了 `stopRecording()` 方法封装锁操作，外部不再直接修改状态。

弹幕线程 `remoteThread` 原来是 busy-wait 模式：`while self._running: await asyncio.sleep(0.5)`。这种模式有两个问题：停止时需要等待最多 500ms；循环本身在每次迭代都有开销。改为 `asyncio.Event().wait()`——协程挂起，零 CPU 消耗，停止信号到达后立即响应。

另外给弹幕客户端加了线性退避重连策略（1 秒起步，每次翻倍，封顶 30 秒）。原来的默认策略是固定 1 秒重试——如果有 32 个窗口同时掉线，B站服务器会瞬间收到 32 个连接请求，然后每 1 秒再来一波。线性退避把重连请求分散到不同时间点，对双方都友好。

### 内存和资源泄漏

- `RecordThread.run()` 里的文件写入如果在循环中抛异常（比如磁盘满了），`self.cacheVideo.close()` 永远不会执行——因为它在循环外面，而异常跳过了它。改为 `try/finally`，无论什么情况文件一定关闭。
- `vtb.csv` 的读写没有用 `with open()`：`open()` 和 `close()` 之间有任何异常都会导致句柄泄漏。两处都改为 `with` 语句。
- `deleteAll()` 清空卡片槽时只把 `roomID` 设成 `'0'`，`CoverLabel` 对象本身从未释放。长时间使用后内存中会堆积大量隐藏的死控件。修复方案是 `hide()` + `deleteLater()` + `clear()`。
- 旧的 `MpvDanmakuRenderer` 的 `QTimer` 没有设置 parent。如果对象被垃圾回收而定时器还在运行，回调会访问野指针。不过这个类已经被整个删掉了。

### 逻辑 Bug

- hex 颜色生成有个数学错误：`int(browser_opacity / 101 * 256)`。除以 101 而不是 100，导致透明度映射偏移。更糟的是 `hex()` 的结果在单字符时（如 `hex(0)` = `"0x0"`，去掉 `0x` 后只剩 `"0"`）会生成无效的 CSS 颜色值——CSS 要求 hex 颜色必须是 6 位或 8 位。改为 `format(round(pct/100*255), '02x')`，确保始终输出 2 位十六进制、alpha 值正确映射。
- `pay.py` 赞助榜排序在列表为空时尝试用未初始化的 `bossNum` 做 `pop()`，直接 `NameError`。加了空列表检查和 `try/except` 保护 money 解析。
- `Slider` 控件把鼠标 X 坐标直接当值用，假设滑块永远 100px 宽。如果窗口缩放导致宽度变化，点击位置和实际值就完全对不上。改为 `int(x / width * maximum)`，按实际宽度比例映射。
- `refreshHacoList` 在信号处理函数里 `disconnect` 自己——快速双击会尝试 disconnect 一个已经断开的信号，`RuntimeError`。加了防重入守卫。
- `checkUpdate` 删掉了 `latest` signal 但 `emit` 调用没删，访问不存在的属性导致后台线程静默崩溃。

## 代码清理

删掉的内容：

- `MpvDanmakuRenderer`（278 行）：被 `DanmakuRenderer` 完全取代的旧弹幕方案。包括 ASS 头模板、弹道分配算法、临时文件管理、字幕重载逻辑——所有这些在新方案中都不再需要。
- `webBrowser.py`（48 行）：被 `QRLoginWidget` 取代，且依赖的 `QWebEngineView` 不在 `requirements.txt` 里，实际上从未可用。
- `sample.py`（120 行）：blivedm 的使用示例，误入生产环境。
- `test.py`（7 行）：一次性的本地调试脚本。
- `latestRemainder` 类和 `latest` signal：定义了但从未实例化和连接的版本检查残留。
- `plugins/` 和 `plugins_lite/` 目录：VLC 插件 DLL 和 Blu-ray JAR 文件，总计超过 200 个文件、约 77MB。项目迁移到 MPV 后完全无用。
- 空桩方法 `copyCache` 和 `newPlayer`：只有 `pass` 和过时的注释。
- 重复定义的 `DISPLAY_RATIOS` 常量：一份在 `danmu.py`，一份在 `config_manager.py`，改为统一从 `danmu.py` 导入。
