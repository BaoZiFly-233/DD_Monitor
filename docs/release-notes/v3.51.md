# DD Monitor v3.51

距离 v2.26 发布已经过去相当长一段时间。这期间项目经历了一次彻底的底层翻新——播放器从 VLC 迁移到了 MPV，弹幕系统全部重写，配置管理从手工作坊升级成了带备份轮转的独立模块，登录方式也从依赖 Qt WebEngine 变成了轻量的二维码方案。下面按模块梳理一下这次更新的内容。

---

## 播放器：从 VLC 到 MPV

VLC 的方案需要对每个直播间启动独立的 VLC 实例，再通过窗口捕获嵌入 Qt。这个架构在 Windows 上问题不少：VLC 的 DLL 插件目录臃肿，不同版本的兼容性难以保证，而且 CPU 占用居高不下。

v3.51 将整个播放链迁移到了 MPV，通过 `libmpv` 的 render API 直接在 `QOpenGLWidget` 上渲染视频帧。好处有几个：

- 不再需要安装 VLC，也不需要携带任何 VLC 插件。Windows 用户只需要一个 `libmpv-2.dll` 文件。
- OpenGL 渲染路径意味着弹幕可以直接用 QPainter 画在视频帧上，不需要依赖 MPV 的字幕系统去做 `sub-reload`——那个方案每次刷新都会导致画面短暂黑屏。
- 硬件解码方面，Windows 上由于 libmpv 与 ANGLE 的兼容问题，我们在 OpenGL 模式下主动禁用了硬解以规避花屏。如果将来 libmpv 修复了这个问题，可以重新打开。

画质切换、音量增强、全局控制这些交互逻辑保持不变，但底层实现全部重写。

---

## 弹幕系统：三件套重构

弹幕是这次改动最大的部分，实际上分成了三个独立模块：

### 渲染器 (danmaku_renderer.py)

旧的方案是在 `danmu.py` 里用 `MpvDanmakuRenderer` 生成 ASS 字幕文件、通过 `sub-add` 加载、每次新弹幕到达时 `sub-reload` 刷新。这个方案有两个致命问题：一是频繁的文件写入和字幕重载造成明显的闪烁；二是 ASS 的 `\move()` 标签依赖 libass 的插值，在不同平台上的表现不一致。

新的 `DanmakuRenderer` 直接在 OpenGL 帧上用 QPainter 绘制弹幕文字。每条弹幕先渲染成 `QImage` 精灵（颜色、描边、阴影一次性到位），然后根据布局引擎分配的轨道位置，在 `paintGL()` 里画出来。精灵有 LRU 缓存，最多保留 512 条，相同的文字和样式只渲染一次。

### 布局引擎 (danmaku_layout.py)

滚动弹幕需要分配轨道——每条弹幕从屏幕右侧进入，以固定速度向左移动，经过整个屏幕后消失。如果有两条弹幕在同一轨道上时间重叠，就会叠加在一起，完全无法阅读。

`RollLayout` 维护了一个轨道池，每条轨道记录当前占用者的过期时间。新的弹幕到达时，优先分配到空闲轨道；如果所有轨道都在使用中，就丢弃这条弹幕（业内标准做法，B站自己也是这么干的）。顶部固定弹幕和底部固定弹幕由 `TopLayout` 和 `BottomLayout` 分别管理，逻辑类似。

轨道的数量由字体大小和显示区域比例自动计算。显示区域比例可以在全局弹幕设置中调节。

### 设置面板 (danmu.py)

新增了 `GlobalDanmuOption`——一个合并了弹幕窗设置和滚动弹幕设置的面板。弹幕窗部分控制悬浮弹幕机的透明度、尺寸、同传过滤关键词、礼物和进入信息的显示策略。滚动弹幕部分控制字体、速度、描边粗细、阴影效果、顶部和底部弹幕的开关。

每个窗口也保留了独立的弹幕机，可以通过窗口控制栏的「弹」按钮在三种模式之间切换：全开（弹幕机 + 滚动）、仅弹幕机、全关。

## 配置管理

旧版本的配置是一团乱麻：`config.json` 的读写散落在主窗口代码的各个角落，格式迁移靠临时补丁，保存时机没有控制，频繁写入磁盘。更糟的是，备份逻辑其实是同时写入四份完全相同的文件——如果某次保存写入了坏数据，四份备份一起坏。

`ConfigManager` 把这些全部收敛到了一个类里：

- 加载时自动做格式迁移，兼容从 v1.x 到 v3.x 的所有配置格式
- 保存有 500ms 的去抖动，用户连续调整参数时不会触发多次磁盘写入
- 备份是真正的轮转：当前配置写入 `config.json` 前，先把旧的 `config.json` 挪到 `备份1`，`备份1` 挪到 `备份2`，`备份2` 挪到 `备份3`。这样不管发生什么，至少有一份上一轮的完整配置可以回滚
- 修复了一个存在已久的 bug：房间列表（`roomid`）在关闭程序后丢失。问题出在 `LiverPanel` 修改了内存中的房间列表，但从未回写到 `config` 字典，导致保存时用的永远是启动时读到的旧值

## 登录

废弃了 `webBrowser.py`（基于 Qt WebEngine 的方案，需要加载 Chromium 内核，启动慢且内存占用大），改用 `login.py` 里的 `QRLoginWidget`。原理很简单：调用 B站 API 生成登录二维码，用 `qrcode` 库渲染成 QPixmap 显示在界面上，后台轮询扫码状态。

Session 信息保存在 `config.json` 中。启动时如果检测到有效的 SESSDATA，会自动调用 B站 API 验证登录状态。Token 过期前会自动刷新。

## 稳定性

这次修了不少"能跑但不该这么跑"的问题：

### 线程安全

`RecordThread`（录制线程）的 `recordToken` 和 `downloadToken` 在跨线程访问时没有任何保护。主线程可以随时把 `recordToken` 设为 `False` 来停止录制，而录制线程在自己的 `run()` 循环里读这个值。在高频录制场景下，这会导致未定义行为。加了 `threading.Lock`，所有跨线程访问都走 `with self._lock`。

弹幕线程 `remoteThread` 原来是 busy-wait：`while self._running: await asyncio.sleep(0.5)`。这种模式在停止时需要 500ms 才能响应。改为 `asyncio.Event().wait()`，停止信号到达后立即响应。

另外给弹幕客户端加了线性退避重连策略（1 秒起步，每次翻倍，封顶 30 秒）。原来的默认策略是固定 1 秒重试——如果有 32 个窗口同时掉线，服务器会瞬间收到 32 个连接请求，然后每 1 秒再来一波。

### 内存和资源泄漏

- `RecordThread.run()` 里的文件写入如果在循环中抛异常，`self.cacheVideo.close()` 永远不会执行。改为 `try/finally` 保证文件一定关闭。
- `vtb.csv` 的读写没有用 `with open()`，代码中的异常会跳过 `close()`。
- `deleteAll()` 清空卡片槽时只是把 `roomID` 设成 `'0'`，`CoverLabel` 对象本身从未释放。长时间使用后内存中会堆积大量死控件。
- `MpvDanmakuRenderer` 的 `QTimer` 没有 parent，对象销毁后定时器可能回调到野指针。

### 逻辑 Bug

- hex 颜色生成有个数学错误：`int(browser_opacity / 101 * 256)`。分母应该是 100 而不是 101，而且 `hex()` 的结果在单字符时（如 `hex(0)` = `0x0`，去掉 `0x` 后只剩 `"0"`）会生成无效的 CSS 颜色值。改为 `format(round(pct/100*255), '02x')`。
- `pay.py` 的赞助榜排序在列表为空时会触发 `NameError`（`bossNum` 未初始化）。
- `Slider` 控件把鼠标 X 坐标直接当值用，假设了滑块永远 100px 宽。如果布局调整导致宽度变化，点击位置和实际值就对应不上。
- `refreshHacoList` 在信号处理函数里 `disconnect` 自己，快速双击会尝试 disconnect 一个已断开的信号，直接 `RuntimeError`。

## 代码清理

删掉的内容大致有这些：

- `MpvDanmakuRenderer`（278 行）：被 `DanmakuRenderer` 取代的旧弹幕方案
- `webBrowser.py`（48 行）：被 `QRLoginWidget` 取代的旧登录方案
- `sample.py`（120 行）：blivedm 的示例代码，误入生产环境
- `test.py`（7 行）：一次性的本地测试脚本
- `latestRemainder` 类和 `latest` signal：定义了但从未使用的版本检查残留
- `plugins/` 和 `plugins_lite/` 目录：VLC 插件的遗骸，约 77MB
- 多处空桩方法（`copyCache`、`newPlayer`）
- 重复定义的 `DISPLAY_RATIOS` 常量

## 构建和 CI

GitHub Actions 工作流会在推送 tag（格式 `v*`）时自动触发：

1. 下载最新的 mpv 运行时文件（从 shinchiro/mpv-winbuild-cmake）
2. 用 PyInstaller 打包 Windows 版本
3. 创建 GitHub Release 并附带打包好的 zip

默认版本号已更新为 3.51。
