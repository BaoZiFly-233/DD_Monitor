# 文档撰写备忘

这份文档写给未来的我（或者任何需要维护这个项目的 AI agent）。记录的是我在为 DD Monitor 写文档时碰到的坑、做出的选择，以及这个项目对文档风格的约定。

---

## 两个核心文档

项目有两份面向读者的文档：`README.md` 和 `docs/release-notes/`。

### README.md

定位：**新用户的第一印象**。一个好的 README 应该在 30 秒内回答三个问题：这是什么、怎么装、怎么跑。

写 README 时我刻意避开了几件事：

- **不要 emoji**。emoji 在 GitHub 上渲染效果不稳定，不同平台的回退字体也不一致。而且严肃的技术项目不需要花里胡哨。
- **不需要版本历史**。README 说的是"现在"，版本历史交给 release notes。
- **项目结构图要跟实际代码同步**。之前 README 里的结构图列出了 `plugins/`、`plugins_lite/`、`webBrowser.py`、`sample.py`——这些文件在 v3.51 里已经删了。每次重构后务必更新结构图。
- **致谢要放在末尾，但不要太后面**。原作者的信息必须保留且放在开头显眼位置，这是基本的开源礼节。魔改维护者的信息可以放在后面但不要喧宾夺主。
- **常见问题不是凑数的**。只放真正被问过的问题。如果某个问题在 Issues 里出现了三次以上，值得放进 FAQ。

### Release Notes

定位：**给已经在用的人看的变更说明**。读者已经知道这软件是干什么的，他们想知道的是：这次更新改了什么、为什么改、对我有什么影响。

写 release notes 时遵循这些原则：

- **按模块叙事，不要按 commit 罗列**。`git log` 输出不是 release notes。按功能域分组（播放器、弹幕、配置、稳定性），每个域里先讲做了什么，再讲为什么这么做。
- **解释"为什么"比"做了什么"更重要**。说"从 VLC 迁移到 MPV"只告诉读者结果，说"VLC 的窗口捕获方案在 16 窗口时 CPU 占用 40%+ 且 DLL 管理混乱"告诉读者动机。好的 release notes 让读者理解设计决策。
- **大版本要体现跨度**。v3.51 距离 v2.26 有一年的 gap，release notes 必须说清楚这段时间发生了什么——原项目停更、紧急修复、然后才是大刀阔斧的重构。
- **bug 修复也要分类**。不是说"修了 28 个 bug"，而是说"线程安全修了 3 个"、"资源泄漏修了 4 个"、"逻辑错误修了 5 个"。归类让读者快速找到自己关心的部分。
- **删除的内容也要记录**。删了 278 行代码、删了 77MB 的 VLC 残留，这些都是信息。
- **前瞻性内容有价值**。在末尾放一个"下一步计划"的列表，让读者知道这个项目还活着、维护者在想什么。但不要承诺时间线——承诺了做不到比不承诺更糟糕。
- **Release notes 不需要致谢**。那个是 README 干的事。

## 写文档时的常见错误

### 错误 1：把文档写成代码注释

文档的读者可能不是程序员，或者不熟悉这个代码库。不要写"修改了 `config_manager.py` 的 `_flush` 方法"，写"配置保存现在有真正的备份轮转，不再同时写四份相同文件"。

### 错误 2：技术细节太多，背景太少

说"用 QPainter 替代 ASS 字幕"是对的，但不够。应该补充"因为 ASS 方案的 sub-reload 在多窗口下产生闪烁，且 libass 的插值算法在不同平台表现不一致"。

### 错误 3：版本号不一致

README 说 v3.50，release notes 说 v3.51，代码里版本号是 3.51——这种事情在手动维护的项目里经常发生。改版本号时需要同时改这些地方：
- `DD监控室.py` 的 `self.versionNumber` 和 `self.versionDisplay`
- `README.md` 里所有提到版本号的地方
- `scripts/build_win.bat` 的默认 `APP_VERSION`
- `.github/workflows/python-app.yml` 的默认参数
- `docs/release-notes/` 下的文件名和内容

### 错误 4：提交信息太敷衍

"update docs" 不是一个好 commit message。好的 commit message 应该：
- 用英文写（Git 生态的约定）
- 前缀标明类型：`docs:`、`fix:`、`feat:`、`refactor:`、`chore:`
- 第一行 50 字符以内概括
- 空一行后写详细说明（如果需要）

例如：
```
docs: 重写 Release Notes，体现 v2.26 到 v3.51 的版本跨度

按模块叙事：播放器迁移、弹幕重构、配置管理、稳定性修复。
添加前瞻性内容。移除 CI 构建说明板块。
```

## 文档文件组织

```
docs/
  release-notes/
    v3.50-魔改版.md    # 上一版本的 release notes（保留，历史参考）
    v3.51.md            # 当前版本
  writing-docs.md       # 这份文件
```

`docs/` 里不放设计文档、计划文档、TODO 列表。那些东西属于 issue tracker 或者项目看板，不属于代码仓库。已经删掉的 `docs/plans/` 和 `docs/live-api.md` 就是反面教材。

## 给下一个 AI agent 的话

如果你被派来维护这个项目，读 README 了解概况，读 `docs/release-notes/` 了解历史变更，读完这份文件再动笔。

几条经验：

1. **先跑一遍软件**。你读到的代码和你运行起来的体验可能是两回事。特别是这个项目——弹幕系统有三套遗留方案，不实际跑一遍你分不清哪套是活的。
2. **修改前先 grep**。很多常量（比如版本号）散落在多个文件里，改一个不改另一个就会引入不一致。
3. **别碰 PSD 文件**。`dd监控室封面.psd` 和 `utils/splash.psd` 是 Photoshop 源文件，不是运行时资源，但用户需要它们来改启动画面。删了还得从 GitHub 下载回来。
4. **release notes 要让人读得下去**。你不是在写 API 文档，你是在跟一个关心这个项目的人聊天。用平实的句子，不要用列表凑字数。
5. **CI 构建一次要 3 分钟**。改 README 和 release notes 不要每改一次就推一次 tag——攒够了再推。
